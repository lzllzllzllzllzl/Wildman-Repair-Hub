# 真实页面人工验收记录

验收日期：2026-07-19  
验收环境：Codex 内置浏览器，`http://localhost:3000`  
说明：以下结果来自真实页面点击、选择、输入和角色切换，不是仅根据 API 或自动化测试推断。

## 场景一：正常自动派单

### 首次真实提交

1. 进入 `/store/report`；
2. 选择光谷店（模拟）、GEL-001；
3. 输入“今天上午开始机器运行时有明显异响，做出来的产品不太成型，已经影响正常出品。”；
4. 点击“提交报修并运行本地 AI”；
5. 页面生成 `WO-20260719-473639`。

页面观察：

- 原始描述逐字保留；
- 摘要为“GEL-001；运行异响、产品成型异常；发生于今天上午；生产影响中。”；
- high 置信度，机械运行异常，证据包含异响与成型/出品异常；
- 命中 R003，责任方为供应商A，P2，待接单；
- 供应商可以接单、开始处理、提交维修结果；
- 门店验收后进入已关闭，状态变化进入时间线；
- 设备履历中可看到该工单。

### 修复后回归

通过 `/demo` 一键载入 `WO-20260719-001` 后再次真实操作：

- 供应商A接单；
- 点击“开始处理”；
- 填写实际故障原因、维修动作和 Demo 配件；
- 提交后进入待验收；
- 门店填写验收意见并通过；
- 状态为已关闭，时间线 5 条；
- 点击“GEL-001 · 查看履历”，设备页显示该已关闭工单。

结果：**通过**。

## 场景二：信息缺失

1. `/demo` 点击“信息缺失并补充”的“一键载入”；
2. 自动进入 `/store/supplements`；
3. 页面显示 FE-002 原文“售卖柜温度一直往上升，现在不知道具体是哪台。”；
4. 页面显示 AI 已识别“温控异常”、缺失“设备编号”、问题“请选择具体故障设备或补充设备编号。”；
5. 此时没有 FE-002 正式工单；
6. 选择 COLD-002，点击“补充并重新分析”；
7. 生成正式工单并进入待接单。

页面观察：

- 原始报修未被补充文本覆盖；
- 当前分析为 v2；
- 展开“分析版本历史”可同时看到：
  - v1：设备待确认；
  - v2：设备为 COLD-002；
- v2 为温控异常、P1、high，风险标签包含温控风险；
- 命中 R009，路由供应商B。

结果：**通过**。

## 场景三：低置信度转人工

1. `/demo` 一键载入低置信度场景；
2. 工单 `WO-20260719-003` 显示原始信息“机器坏了，没反应。”；
3. AI 显示 low 置信度、无法启动、紧急度待确认；
4. R010 明确说明低置信度禁止自动派单；
5. 切换维修管理，从 `/manager` 再进入公共 `/orders/order-003`；
6. 角色仍保持“维修管理人员”，未再错误重置为门店；
7. 不填写人工原因点击提交，页面保持待人工确认；
8. 填写类别、P2、供应商A和决策原因后提交。

页面观察：

- 原始信息、AI建议、规则结果、人工决策同时可见；
- 人工提交前没有最终责任方；
- 提交后进入待接单；
- 时间线记录类别、等级和责任方修改前后值及人工原因。

结果：**通过**。

## 场景四：P1 超时

1. `/demo` 一键载入 `WO-20260719-004`；
2. 详情显示 P1、待接单、Demo 模拟 SLA 和秒级倒计时；
3. 切换运营，进入 `/admin/demo-control`；
4. 选择 `WO-20260719-004`；
5. 点击“模拟即将超时”，详情倒计时进入风险区；
6. 点击“模拟已经超时”；
7. 切换维修管理，进入 `/manager`；
8. 工单出现在优先处理区域，状态为超时未接单；
9. 通知区域出现维修管理“超时升级”和运营“P1超时升级”；
10. 打开工单，不填写改派原因无法提交；
11. 选择供应商A并填写原因后重新派发。

页面观察：

- 状态按“超时未接单 → 待人工确认 → 待接单”变化；
- 原责任方供应商B、新责任方供应商A和原因进入时间线；
- 改派后新的 SLA 重新计时。

结果：**通过**。

## 场景五：验收不通过

1. 重置后进入 `/store/acceptance`，打开 `WO-20260719-007`；
2. 页面展示维修人员实际诊断和维修动作；
3. 不填写“不通过原因”点击按钮，状态保持待验收；
4. 填写“设备可以运行，但前场温度仍高于 Demo 验收标准。”后提交；
5. 状态进入返修中，返修次数显示为 1；
6. 切换供应商，在任务中心选择设施供应商D；
7. 同一工单出现在返修列；
8. 点击“接受返修”，进入处理中；
9. 空维修原因/动作不能提交；
10. 填写第二次维修结果后提交，进入待验收；
11. 切换门店，填写二次验收意见并通过。

页面观察：

- 最终状态为已关闭；
- URL 和工单 ID 始终是 `order-007`；
- 时间线共 6 条，包含返修与二次维修；
- 没有创建无关联新工单。

结果：**通过**。

## 权限和非法操作

| 检查 | 页面/测试结果 |
|---|---|
| 错误供应商操作他方工单 | 页面显示“不可操作其他责任方工单”，无操作按钮；服务端校验责任方 ID |
| 人工原因空值 | 浏览器 required 拦截，状态不变 |
| 验收不通过原因空值 | 浏览器 required 拦截，状态不变 |
| 维修原因或动作空值 | 浏览器 required 拦截，状态不变 |
| 门店修改责任方/等级/SLA | 页面无入口；角色测试拒绝 |
| 门店或供应商取消 | 页面无入口；角色测试拒绝 |
| 非法状态流转 | 状态机测试返回“非法状态流转” |
| 已关闭工单继续维修 | 无操作入口；状态机没有任何关闭后流转 |
