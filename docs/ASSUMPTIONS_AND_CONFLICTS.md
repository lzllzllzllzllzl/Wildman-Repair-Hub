# 假设、冲突与证据边界

## 使用原则

Deep Research 只用于界定企业已确认事实；系统范围以 v0.1 和本次 Demo V1 要求为准；页面/交互/字段以 v0.2 为准；模拟字段与自动化以 v0.3 为准。本文件中的处理决定仅服务于产品验证，不代表野人先生真实制度。

## 已识别冲突与处理

| 项目 | 文档差异或未知 | Demo V1 处理 |
|---|---|---|
| 真实维修现状 | 公开资料不能证明微信/电话报修、信息缺失、无工单或无 SLA | 界面不声称这些是企业现状；只把它们作为待验证问题 |
| 状态数量 | v0.1/v0.2/v0.3 包含草稿、预约、等待配件等较完整状态；本次要求冻结 10 个核心状态 | 业务状态只使用本次指定的 10 个；预约等作为字段/动作，不增加状态 |
| P1 完全停产 | v0.2 R03 可允许自动派单；高风险规则又要求人工 | 食安、人身、电气风险强制人工；只有“业务中断但无安全风险”的 P1 可按唯一规则派单 |
| 缺设备的工单 | WorkOrder 要求 assetId；缺设备场景在补充前无法满足 | 先保存 FaultEvent，补充并识别设备后才创建 WorkOrder |
| 低置信度工单 | 必须进入管理队列，但责任方尚未确定 | 创建待人工确认 WorkOrder，最终责任方允许暂时为空；人工确认时补齐 |
| 接单状态 | v0.2 有“已接单”，本次核心状态未列出 | 接单后直接进入“处理中”，acceptedAt 保留接单时间 |
| SLA 数值 | v0.1/v0.2/v0.3 使用 1/2/5 分钟；本次要求 60/120/300 秒 | 完全等价，数据库统一按秒存储 |
| 真实供应商/型号 | Deep Research 只确认存在协同角色，未公开名称、绑定和保修 | 所有名称使用“设备供应商A（模拟）”等，并显示 Demo 标签 |
| 自动超时 | 本地 Demo 无常驻任务调度器 | 每次读取工作台时执行到期检查；另提供“模拟已超时”按钮保证演示稳定 |
| 附件 | V1 要求本地优先且无需外部服务 | 只保存演示附件 URL 文本，不建设对象存储或真实上传 |
| AI 分析历史 | v0.3 要求每次分析新增记录；本次 P0 实体未列 AIAnalysis | FaultEvent 保留当前结果与 analysisVersion，状态日志记录重跑；独立版本表列为后续扩展 |

## Demo 假设

- 门店、设备、责任方、保修、区域与联系方式均为模拟；
- 设备类别用于验证流程，不代表企业最终资产分类；
- 30 天重复故障窗口为可配置 Demo 假设；
- 角色切换器只模拟身份，不是生产认证授权；
- 供应商接单、通知和 SLA 都在本地数据库内模拟；
- 不提供拆机、接电、复位、制冷剂等危险操作建议。

## 仍需企业验证

- 真实资产编号、设备类别、保修与供应商绑定；
- 维修管理组织与各角色权限；
- 接单、到店、修复、验收 SLA 与暂停规则；
- 食品安全、温控、停机与复工验收标准；
- 直营/加盟门店责任和费用边界；
- 飞书身份、外部供应商权限、开放平台凭证与消息目标。
