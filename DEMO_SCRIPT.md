# 比赛 Demo 演示脚本

演示前执行：

```bash
npm run demo:setup
npm run dev
```

打开 `/demo`。开场只说一次：**当前系统全部门店、设备、供应商、SLA和工单均为 Demo 模拟，不代表野人先生真实制度或运营情况。**

## 2 分钟标准版

| 时间 | 页面 | 角色 | 按钮 / 操作 | 讲解核心价值 |
|---|---|---|---|---|
| 0:00–0:12 | `/demo` | 门店 | “正常自动派单”→“一键载入” | 门店用自然语言表达现场事实，不需要理解维修分类 |
| 0:12–0:35 | `/orders/order-001` | 门店 | 指向四块决策卡；展开“查看规则命中依据” | AI 把非结构化信息变成可流转数据；R003 负责确定性责任路由；人工边界独立 |
| 0:35–0:55 | `/supplier` → 工单详情 | 供应商A | “接单并进入处理”→“开始处理” | 三方围绕同一工单协作，接单和处理均进入时间线 |
| 0:55–1:15 | 同上 | 供应商A | 填写“传动组件运行异常（Demo）”“完成模拟检修与运行验证”→“提交维修结果” | 实际诊断由维修人员填写，AI 不代替专业判断 |
| 1:15–1:35 | 同一工单 | 门店 | “验收通过并关闭” | 未经验收不能关闭，门店对最终结果负责 |
| 1:35–1:50 | `/assets/asset-gel-001` | 门店 | 点击“GEL-001 · 查看履历” | 闭环数据沉淀到设备维修履历 |
| 1:50–2:00 | `/admin` 或 `sla-timeout.png` | 运营 | 指向异常指标或备用截图 | SLA 监控把异常推给维修管理和运营；指标均为 Demo 模拟 |

叙事必须连成一句：

> 门店自然语言报修 → AI 结构化 → 规则定责 → 三方协同 → SLA 监控异常 → 门店验收 → 设备履历。

不要逐字段讲解。

| 时间 | 页面 | 角色 | 按钮 / 操作 | 讲解核心价值 |
|---|---|---|---|---|

## 演示失败备用路径

### 状态被前一次演示改变

返回 `/demo`，点击对应场景的“一键载入”。每次载入都会恢复确定性种子数据。

### P1 倒计时自然到期

切换运营 → `/admin/demo-control` → 选择 `WO-20260719-004` → “恢复正常时间”，再按需要点击“模拟即将超时”或“模拟已经超时”。

### 某一步输入来不及

不要绕过必填校验。直接进入下一条场景，使用 `screenshots/order-timeline.png`、`screenshots/acceptance.png` 或 `screenshots/asset-history.png` 讲解闭环。

### 页面或本地服务异常

按以下顺序展示备用截图：

1. `store-report.png`；
2. `ai-analysis.png`；
3. `manager-review.png`；
4. `supplier-task.png`；
5. `sla-timeout.png`；
6. `acceptance.png`；
7. `asset-history.png`；
8. `operations-dashboard.png`。

### 演示表达提醒

- 说“Demo 模拟规则/供应商/SLA”，不说“野人先生当前就是这样执行”；
- 说“AI 建议”，不说“AI 自动诊断并决定责任”；
- 说“规则路由”，不说“模型自由选择供应商”；
- 说“维修人员填写实际原因”，不把 AI 摘要当维修结论；
- 主讲业务价值，不逐字段读页面。
