# 野人先生 AI 设备维修协同系统需求说明书 v0.1

> 文档性质：第一阶段 Demo / PoC 系统需求基线  
> 日期：2026-07-19  
> 说明：本文中的门店、设备、供应商、SLA、规则和案例均可使用模拟数据，用于验证产品逻辑和技术链路，不代表野人先生当前真实制度或运营数据。

## 0. 核心结论

本项目采用“统一系统底座 + 可配置业务模块 + AI 能力模块”的建设思路。

系统先建立统一的门店、设备、供应商、规则、故障事件、工单、状态和维修记录数据底座，再增加：

1. AI 报修信息理解；
2. 缺失信息识别与追问；
3. 故障分类和业务影响建议；
4. 确定性责任路由；
5. SLA 与通知自动化；
6. 门店—维修部—供应商三方工单协同；
7. 门店验收、返修与关闭；
8. 设备履历和管理分析。

首版目标不是建设完整 EAM/CMMS，而是完成一条可演示、可验证、可扩展的维修协同闭环。

## 1. 产品定位

### 1.1 产品名称

**野人先生 AI 设备维修协同中枢**

### 1.2 一句话定位

面向门店、维修管理人员和设备供应商，通过统一设备主数据、统一维修工单、确定性规则与 AI 信息理解，完成从设备故障发现到门店验收关闭的三方维修协同闭环。

### 1.3 首版必须证明

- 门店能够选择设备并提交报修；
- AI 能够把自然语言转成结构化故障信息；
- 系统能够识别关键字段缺失并要求补充；
- 规则能够结合设备、故障、保修、区域和供应商推荐责任方；
- 低置信度、规则冲突和高风险工单能够转人工；
- 系统能够创建唯一工单并派发；
- 责任方能够接单、预约、维修和提交结果；
- 系统能够记录状态、责任人和关键时间戳；
- 门店能够验收或退回返修；
- 工单关闭后能够形成设备履历；
- Demo 能稳定演示正常、信息缺失、低置信度和超时案例。

## 2. 项目边界

### 2.1 当前范围

聚焦以下主链路：

```text
故障发现
→ 设备选择与报修
→ AI 信息结构化
→ 信息完整性检查
→ 缺失信息补充
→ 故障与业务影响建议
→ 责任方和 SLA 匹配
→ 人工复核或自动派发
→ 接单
→ 远程处理或预约上门
→ 维修执行
→ 门店验收
→ 返修或关闭
→ 设备履历与分析
```

### 2.2 当前不做

- 设备采购和完整资产生命周期；
- 预防性维护和预测性维修；
- IoT 实时监测；
- 完整备件库存和采购；
- 维修费用结算；
- 供应商合同审批；
- 设备报废与更新；
- AI 独立专业诊断；
- AI 输出拆机、接电、制冷剂等高风险操作；
- 生产级系统上线；
- 与企业真实 ERP、采购、财务系统深度集成；
- 承诺真实 ROI 或维修效率提升比例。

## 5. 角色与权限

### 5.1 门店

可操作：

- 选择设备；
- 发起报修；
- 上传文字、语音、图片和视频；
- 补充信息；
- 查看状态；
- 配合预约；
- 验收通过或不通过；
- 查看本门店历史工单。

不可操作：

- 修改供应商、SLA 和路由规则；
- 修改其他门店数据；
- 直接关闭未验收工单；
- 自行变更最终责任方。

### 5.2 维修管理人员

可操作：

- 查看待人工确认、拒单、超时和异常工单；
- 修改设备、故障类别、紧急度和责任方；
- 转派、升级、取消和重开；
- 处理规则冲突；
- 管理基础规则；
- 查看全局数据。

所有人工修改必须记录操作者、时间和原因。

### 5.3 供应商 / 工程师

可操作：

- 接单；
- 拒单并选择原因；
- 请求补充；
- 选择远程处理或上门；
- 填写预约时间；
- 更新到店和维修状态；
- 填写实际故障、维修动作和配件；
- 上传结果；
- 提交待验收。

### 5.4 运营管理员

- 管理门店、设备、供应商和人员；
- 配置权限、规则和通知对象；
- 查看超时、返修、重复故障和供应商表现；
- 查看和导出管理数据。

## 7. 端到端流程

### S01 选择设备并报修

门店扫码或从设备列表选择设备，输入文字、语音、图片、发生时间和业务影响。

输出：故障事件。

### S02 校验设备归属

规则检查设备是否属于当前门店。

- 匹配：进入 AI 分析；
- 不匹配：进入“设备待确认”。

### S03 AI 信息结构化

AI 提取：

- 设备类型；
- 故障现象；
- 时间；
- 位置；
- 生产和营业影响；
- 风险标签；
- 缺失字段；
- 故障类别建议；
- 紧急度建议；
- 置信度和依据。

### S04 信息完整性检查

确定性规则判断 P0 字段是否完整。

- 完整：进入路由；
- 不完整：生成补充任务。

### S05 门店补充

系统定向要求门店补拍铭牌、补充设备号、故障图片或业务影响。补充后重新分析。

### S06 责任路由

系统读取：

- 设备；
- 故障类别；
- 保修状态；
- 服务区域；
- 供应商绑定；
- 紧急度；
- 规则优先级。

输出：推荐责任方、路由理由和 SLA。

### S07 人工复核

以下情况必须转人工：

- AI 置信度低；
- 设备无法识别；
- 多个责任方同时命中；
- 无规则命中；
- 涉及食安、安全或重大风险；
- 数据矛盾；
- 供应商无效。

### S08 创建并派发工单

系统创建唯一工单，记录责任方、派发时间、SLA 和路由原因。

### S09 接单

责任方可以：

- 接单；
- 拒单；
- 请求补充；
- 选择远程处理；
- 选择预约上门。

### S10 预约与维修

责任方填写预约、工程师、预计时间、实际故障、维修动作、配件和结果。

### S11 门店验收

- 通过：关闭；
- 不通过：填写失败项和证据，进入返修。

### S12 关闭和沉淀

系统关闭工单，写入设备履历，生成待审核知识摘要，并计算流程指标。

## 8. 状态机

### 8.1 主状态

```text
草稿
→ AI 分析中
→ 待补充
→ 待人工确认
→ 待派发
→ 待接单
→ 已接单
→ 待预约
→ 远程处理中 / 待上门
→ 维修中
→ 待验收
→ 已关闭
```

### 8.2 异常状态

```text
设备待确认
拒单
超时未接单
等待门店补充
等待配件
暂停
无法修复
返修中
已取消
已重开
```

### 8.3 原则

- 一次故障一个主工单；
- 返修和重开保留原记录；
- 每次责任转移记录原因和时间；
- 未验收不得关闭；
- AI 不得关闭工单；
- 业务状态、责任状态和 SLA 状态分开；
- 关闭后再次出现同类故障时关联原工单。

## 9. 功能需求

### M01 门店智能报修

- 支持扫码、编号和列表选设备；
- 支持文字、语音、图片和可选视频；
- 支持填写生产、营业、温控和安全影响；
- 提交后返回事件编号和处理状态；
- 可通过消息卡片继续补充。

### M02 AI 信息处理

AI 必须输出固定结构：

```json
{
  "standard_summary": "",
  "device_type": "",
  "symptoms": [],
  "occurred_at_text": "",
  "production_impact": "",
  "business_impact": "",
  "risk_tags": [],
  "missing_fields": [],
  "fault_category_suggestion": "",
  "priority_suggestion": "",
  "responsibility_explanation": "",
  "confidence": "high|medium|low",
  "evidence": []
}
```

要求：

- 未知信息输出“不确定”；
- 不编造设备、供应商、保修或故障原因；
- AI 只输出建议，不做最终诊断；
- 低置信度不得自动派单；
- 涉及高风险时只提示停止自助操作并转人工。

### M03 人工复核工作台

展示：

- 低置信度；
- 设备不匹配；
- 规则冲突；
- 高风险；
- 供应商拒单；
- 超时工单。

人工可修改设备、故障类别、紧急度和责任方，修改必须留痕。

### M04 路由和派单

规则支持：

- 设备类别和型号；
- 故障类别；
- 保修状态；
- 服务区域；
- 供应商绑定；
- 门店类型；
- 紧急度；
- 规则优先级。

系统必须展示可解释的路由理由。

无唯一结果时转人工。

### M05 供应商任务台

供应商可：

- 接单；
- 拒单并选择原因；
- 请求具体字段补充；
- 选择远程或上门；
- 填写预约；
- 更新到店和维修状态；
- 提交维修结果。

### M06 预约和维修

记录：

- 预约时间窗口；
- 工程师；
- 出发、到店、开始、完成时间；
- 实际故障原因；
- 维修动作；
- 配件；
- 是否一次修复；
- 结果附件。

### M07 验收和返修

维修完成后自动通知门店。

验收表按设备类型展示验收项。

验收不通过必须填写原因，工单进入返修并返回原责任方。

只有验收通过后才能关闭。

### M08 SLA 和通知

支持配置：

- SLA 等级；
- 起止点；
- 工作时间；
- 暂停和恢复；
- 即将超时阈值；
- 升级对象。

Demo 可配置：

- P1：1 分钟；
- P2：2 分钟；
- P3：5 分钟。

所有界面明确标注“Demo 模拟时长”。

通知包括：

- 新报修；
- 待补充；
- 待人工确认；
- 派单；
- 即将超时；
- 已超时；
- 拒单；
- 预约；
- 维修完成；
- 待验收；
- 验收不通过；
- 关闭。

### M09 设备履历和知识

设备页展示：

- 历史工单；
- 故障类别；
- 维修次数；
- 返修次数；
- 最近维修时间；
- 重复故障。

AI 可生成维修知识摘要草稿，但必须人工审核后才能发布。

### M10 运营看板

首版展示：

- 工单总数；
- 状态分布；
- 设备故障分布；
- 信息补充率；
- AI 低置信度率；
- 一次定责成功率；
- 接单超时数量；
- 闭环率；
- 验收一次通过率；
- 返修数；
- 重复故障数。

## 10. 路由规则

优先级：

```text
安全/食品安全硬规则
> 设备唯一绑定和保修规则
> 设备类别与故障类别
> 区域服务范围
> 默认责任方
> 人工复核
```

示例：

```text
设备 = Gelato 制作设备
且 保修内
且 绑定供应商有效
→ 责任方 = 绑定供应商
```

```text
设备 = 低温设备
且 风险包含温控/食品安全
→ 紧急度 = P1
→ 通知维修管理和运营
```

```text
AI 置信度 = 低
→ 禁止自动派单
→ 状态 = 待人工确认
```


## 12. 异常流程

### 12.1 设备无法识别

```text
无法匹配
→ 请求扫码/上传铭牌
→ 仍无法识别
→ 维修管理创建临时设备
→ 人工定责
```

### 12.2 供应商拒单

```text
拒单并选择原因
→ 待人工确认
→ 维修管理复核
→ 转备选供应商或内部处理
```

### 12.3 无人接单

```text
待接单
→ SLA 到期
→ 自动提醒
→ 自动升级
→ 重新派发
```

### 12.4 等待配件

```text
选择等待配件
→ 填写配件和预计时间
→ 记录暂停原因
→ 配件到位
→ 恢复维修
```

### 12.5 验收不通过

```text
门店填写失败项和证据
→ 标记返修
→ 返回原责任方
→ 再次维修
→ 再次验收
```

### 12.6 重复故障

```text
新事件
→ 匹配近期历史
→ 标记疑似重复
→ 维修管理确认
→ 必要时升级复盘
```

## 16. 实施顺序

### 第一步：数据底座

完成接单、拒单、预约、处理和状态更新。

### 第五步：SLA 和验收

完成超时提醒、维修结果、门店验收、返修和关闭。

### 第六步：测试与 Demo

准备 20—30 条模拟数据，修正 Prompt 和规则，固化 3—4 个案例并录制备用视频。
